Lucas Ackman

Overview

My name is Lucas Ackman, I am a software engineer specialized in video game development. My passion lies in creating engaging gaming experiences for players. With a strong background in software engineering, I understand the development process of game development. However I also offer a unique perspective on topics due to the personal journey I've taken to get where I am now.

Having grown up with a disability that necessitated the use of crutches, computers became my primary form of entertainment. This experience fueled my desire to share the same joy and entertainment I found in games with others, regardless of their life circumstances.

My skill set encompasses various areas, including software engineering, SOLID design principles, gameplay programming, game design, level design, and web development. This broad expertise enables me to contribute to different aspects of game development and tackle diverse challenges.

Throughout my career, I have worked on a range of projects with varying scopes. I have actively participated in numerous game jams, where I thrived in fast-paced and collaborative environments. Additionally, I have also worked on longer-term projects, allowing me to gain a deeper understanding of the development lifecycle.

One project that I am particularly proud of is a package I have been developing for Unity projects. This package is designed to be universally useful in all Unity projects, offering essential classes such as an input system, serialization system, and other commonly utilized features. It serves as a testament to my dedication to creating efficient and reusable solutions using the SOLID design principles.

To explore my portfolio and discover more about my projects, please use the 'open' command followed the project you want to explore.

Unity Kickstarter Package

When I began my senior year at the University of Wisconsin, Whitewater, I started a new project to serve as a starting point for all of my future Unity Projects. I later labeled this project as my Kickstarter Package, as it contained essentials needed to kickstart most if not all of my projects, such as input management, serialization, event management, and more.

Data Types: I wanted to create scriptable objects to contain data such as floats, ints, or other common data types. This was primarily to decouple systems that needed to share access to some values such as player health.

Inputs: I crafted an enhanced input system using the Unity Input System for improved performance, employing events. Throughout development, I refined the system: I began with the scriptable objects above for decoupling, but later adopted delegates and integrated Actions. This approach led to an efficient input solution.

Events: I managed to develop two separate event solutions. In my initial approach, I used scriptable objects as event buses, while the later involved creating an event manager which controlled every event that was involved with it. Each event bus functions as its own event entity, while the event manager serves as a centralized static class, housing a comprehensive list of all events.

Animations: I utilized an abstract class as an Animation Manager, streamlining animation implementation. Inheriting classes gained access to protected methods for animator control through animation state names, bypassing the need for complex animation trees. I also provided options to modify an animator's parameters using dedicated methods if developers prefer the animation trees.

Serialization: I designed an abstract class to serve as a serialization manager, streamlining data saving and loading through delegates. This approach mandated data types to implement TryParse and ToString methods via an interface. ToString would print out any information required to reconstruct it, while TryParse instructs on how to reconstruct the data type using that information.

I also brought in some smaller additions: object categorization and player data management. Object categories used scriptable objects rather than strings to flexibly group GameObjects. The player class held player info, primarily their input type, making sure everyone gets the correct input subscriptions in local multiplayer games.
